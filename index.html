Agentic AI for consumers

Building personal assistants that act on the world to get arbitrary tasks done — with four demos.

Notes & demos · June 2026

All of the demos below stress a core change in consumer-AI experiences, in particular: agentic AI for consumer use-cases. This is an expansion from the dominant model of consumer AI today (e.g., in ChatGPT), where AI is an advanced search tool that automates information-gathering to answer questions (i.e., “read-tasks”, “deep research”). Instead, with the right setup, we can now enable AI as a personal assistant that actually acts on the world to get arbitrary tasks done for the user (i.e., “write-tasks”, “tool calling in the real world”).

Regarding agentic AI at large, we’re seeing incumbents like Anthropic, OpenAI, and GDM capitalize on this in coding and some broader enterprise use-cases today. But there’s a large open vertical of consumer-facing agentic AI that is largely untapped. Interestingly, AI companies like Meta may be uniquely positioned to lead here by building on their existing infrastructure (Instagram, Facebook, WhatsApp).

The demos

All the demos show capabilities of how flawed-flowed can act as a personal assistant on the user’s behalf to get tasks done. As a reminder, many of these tasks cannot be completed by gathering information from the Internet (“read” operations). Instead, the model needs to “write”, using the Internet to actually enact some change on the world (booking a reservation, communicating with a business, sharing information to friends) on the user’s behalf.

What’s still to do?

Realizing the above requires several core capabilities on both the modeling and larger infrastructure sides.

Robustness and reliability. Most importantly, we can only enable the above if models can complete tasks at the same level of competence as a human executive assistant. Unlike “superintelligence” tasks (e.g., discovering scientific and mathematical breakthroughs) where the expectation is failure — we care about recall not precision, and this ironically makes deploying models here more palatable and success easier in some ways1 — being superhuman at “everyday” tasks means satisfying a different bar for success, where the expectation is success.

Efficiency. Once models can robustly and reliably complete tasks, we also want models to be able to complete tasks quickly. This is both motivated by a better user experience (most would prefer to get a task done sooner rather than later), and better costs (saving tokens and wall-clock time leads to lower costs on our end).

One potentially subtle point is that users may actually place more of a premium on robustness / reliability versus speed than initially expected. This is because even if a task takes a bit longer than users completing it themselves, there’s value in shifting the cognitive load of the task to an assistant (they can “set it and forget it”). Importantly, this cognitive load may only lift if the user can trust the assistant (i.e., the task is done robustly and reliably). But if that’s the case, then it may suffice for a task to be done within a certain reasonable time window, versus trying to drive efficiency to completing tasks in the shortest time possible.

Appendix: design desiderata

Simplicity and generality. Similar to how there is a shift from tool-calling MCPs to terminal-only code execution, we design tool execution around simple yet generalizable infrastructure and primitives.

Rather than require specific APIs and site-specific tools (e.g., for WhatsApp, Telegram, Instagram, etc.), we simply start with browser-use as a universal primitive. This gives us a “basis” to capture all the things a human can do, without needing to write up specific tools to support site-specific features and functionality. For now we simply wrap Playwright via the playwright-cli package, which provides some post-processing of the DOM to make it more readable and accessible for LLMs. Following this approach, we could also build our own setup on top of even lower primitives, e.g., the Chrome DevTools Protocol (CDP). We also note that as the coding models get better, they may be able to code up their own robust APIs for any website.

  1. As an aside, relatably there seems to be a strong correspondence between the cost of failure in a task and the level of success. We made rapid progress in math and code because the cost of checking for correctness was low, and the cost of failure was epsilon — not even in terms of annotation effort, but the cost (and ability even) to reverse any harmful impacts of failure (a wrong math proof maybe hurts less than placing a pizza order to the wrong place). But across both superintelligence and everyday tasks, we see less progress in tasks where the cost of failure is high. For example, consider developing new cures for longstanding illnesses: a superintelligent task with relatively less AI progress, where even if the model is able to generate ideas at the same level of insight + creativity as a human expert, the cost of verifying these ideas through clinical trials is high enough that we currently desire more conviction to fully validate success and make progress. ↩
